## Answer

- **File:** `server/services/sandbox.ts`
- **Entry function:** `executeInSandbox()` at `server/services/sandbox.ts:289` (with `createSandboxContext()` at line 178 building the context)
- **Isolation primitive:** Node.js built-in **`vm` module** — specifically `vm.createContext()` (line 273) plus `vm.Script.runInContext()` (line 298)

How it locks things down:
- `codeGeneration: { strings: false, wasm: false }` blocks `eval` and `new Function`
- A `require()` shim allows only a whitelist of safe builtins (crypto, buffer, url, util, events, stream…) and explicitly rejects 50+ dangerous modules (`child_process`, `fs`, `net`, `process`, `vm`, `worker_threads`, http/https, plus their `node:` variants)
- `process`, `global`, `globalThis`, `eval`, `Function` are set to `undefined` in the context
- 5000ms default timeout with `breakOnSigint: true`

Caller: `server/services/code-executor.ts` (lines 111–128 and 385–398) for Code and Utilities workflow nodes.

Caveat the file itself notes: `vm` is not a hard security boundary against adversarial code — `isolated-vm` or container isolation would be needed for true multi-tenant untrusted execution.
